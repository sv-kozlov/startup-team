# PRD Template

Use this as a structure, not as mandatory bulk. Keep sections that help the next team work.

## Header

- Product or feature name
- Status: Draft / In Review / Approved
- Author
- Stakeholders
- Date created
- Last updated
- Version
- Language: Russian / English / Both

## Executive Summary

- One-line description
- Problem being solved
- Proposed solution
- Primary users
- Primary success metric
- Target launch or delivery window

## Problem Statement

### Current Situation

Describe the current user or business pain.

### Impact

- User impact
- Business impact
- Operational or technical impact, if relevant

### Why Now

Explain urgency, opportunity, risk, or dependency.

## Goals and Non-Goals

### Goals

- Business goals
- User goals
- Product goals

### Non-Goals

List what is explicitly excluded from this PRD.

## Users and Personas

For each primary persona:

- Role or segment
- Goals
- Pain points
- Context of use
- Constraints or accessibility needs

## Scope

### MVP / P0

Must be present for initial value.

### P1

Important but can follow MVP.

### P2 / Future

Useful later, not needed for initial release.

## User Stories and Acceptance Criteria

Group stories by epic or workflow.

For each story:

- Title
- Priority: P0 / P1 / P2
- User story or job story
- Acceptance criteria
- Edge cases
- Dependencies
- Notes for design, analytics, QA, or engineering

## Functional Requirements

Describe behavior in product language:

- main workflows;
- states and transitions;
- permissions and roles;
- data inputs and outputs;
- errors and empty states;
- notifications;
- admin or operational controls.

## Non-Functional Requirements

Use only relevant sections:

- Performance
- Reliability
- Security
- Privacy and data retention
- Compliance
- Accessibility
- Localization
- Compatibility
- Observability and auditability

## Design and UX Requirements

- Key user flows
- Required screens or states
- Content and messaging constraints
- Accessibility expectations
- Design system or brand constraints
- Required prototypes, mockups, or assets

## Technical Considerations

Keep this high-level unless the user asked for technical detail:

- integrations;
- APIs;
- authentication and authorization;
- data model or storage implications;
- migration needs;
- platform constraints;
- external services;
- AI model/tool dependencies, if relevant.

## Success Metrics

For each metric:

- Name
- Definition
- Baseline
- Target
- Measurement method
- Owner or source system
- Time window

Include analytics events when known.

## Rollout and Phasing

- MVP release plan
- Post-MVP phases
- Experiment or staged rollout
- Feature flags or fallback plan
- Launch communications

### Dependencies

- Teams
- Systems

## Open Questions

Use open questions instead of inventing facts.
