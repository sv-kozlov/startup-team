# Project Planning

## Purpose

Turn delivery objective, constraints, and team inputs into a manageable project plan.

## Use When

- A project, release, or implementation needs planning.
- Milestones, workstreams, assumptions, and owners must be made explicit.
- A plan must reflect scope, dependencies, risk, and decision cadence.

## Inputs

- Objective, scope baseline, stakeholders, teams, constraints, estimates, dependencies.

## Workflow

1. Define delivery objective and planning horizon.
2. Identify workstreams, milestones, owners, dependencies, and constraints.
3. Separate committed dates from assumptions.
4. Add risks, decisions, and readiness checkpoints.
5. Hand off content gaps to the owning role.

## Outputs

- Project plan.
- Milestone map.
- Assumption and dependency list.

## Boundaries

- Does not create product priority or backlog scope.
- Does not invent engineering estimates or resource availability.
- Does not own technical, QA, or DevOps content decisions.
