# User Story and Use Case Modeling

## Purpose

Represent user goals and business scenarios in a form that product, system analysis, UX/UI, QA, and delivery teams can refine and implement.

## Use When

- Stakeholder needs must become user stories or use cases.
- A workflow has multiple actors, alternate paths, or exceptions.
- Acceptance and scenario coverage must be clarified before delivery.

## Inputs

- User goals, roles, personas, process steps, business rules, constraints, or feature ideas.
- Existing backlog items, use cases, journey notes, or stakeholder examples.
- Known acceptance signals and exclusions.

## Workflow

1. Identify user goal, actor, trigger, outcome, and business value.
2. Choose story format for backlog work or use-case format for richer scenario logic.
3. Capture main flow, alternate flows, exceptions, preconditions, and postconditions.
4. Link business rules and acceptance criteria to each scenario.
5. Mark technical details for System Analyst and interaction details for UX/UI.

## Outputs

- User stories.
- Use case brief.
- Scenario and exception list.
- Business-facing acceptance logic.

## Boundaries

- Does not own backlog priority.
- Does not define UI layouts or interaction design.
- Does not define technical solution details.
